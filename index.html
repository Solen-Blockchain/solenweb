✦ AGENT-NATIVE

AI agents, enforced by consensus.

Grant an agent a session key with a spending budget, an allowlist of contracts, and an expiry — and the chain itself enforces every limit. No custodial middleman, no exposure of your main key. The agent literally cannot widen its own cage.

1
Grant
From your wallet, mint a session key and scope it: a lifetime budget, per-op cap, an allowlist of contracts and methods, and an expiry height — with optional sub-call lockdown. You sign the grant; the agent receives only the scoped key.
2
Spend
The agent transacts autonomously through the SDK, gasless via a paymaster. Every operation is validated against its limits in consensus — anything over budget, off the allowlist, or past expiry is rejected on-chain.
3
Revoke
Remove the agent at any time in one click, and watch its remaining budget live on-chain. Keys also expire automatically at the block height you set, so an agent’s authority is always bounded.
agent-sdk.ts 
import { SolenAgent } from "@solen/agent-sdk";

// Owner grants a scoped, budgeted key (from the wallet)
const agent = await SolenAgent.create({
  rpcUrl, chainId, owner,
  sessionSecretKey,   // budget · targets · expiry
});

// The agent acts — gasless, only within its limits
await agent.transfer(recipient, 1n * SOLEN);
// ↳ rejected on-chain if it exceeds the budget
Consensus-enforced limits
Lifetime budget, per-op cap, allowed targets, and an expiry height — enforced in consensus, not a backend you trust.
Gasless by design
Paymaster sponsorship lets agents transact with zero native balance. Fund the work, not the wallet.
Scoped session keys
Grant and revoke agents in a click. The agent holds a scoped key — never your main one — and can’t widen it.
Sub-call lockdown
Optionally bound the entire call tree, so operations stay confined to exactly the contracts you approve.
✦ QUANTUM-READY

Ready for the post-quantum era.

Opt in to NIST-standardized post-quantum signatures and protect any account against future quantum computers — today, with one command and no change of address. Classical signing stays the default.

# Upgrade any account to post-quantum auth — same address
solen key quantum-upgrade myaccount

# ...or hybrid: Ed25519 + ML-DSA-65, both required
solen key quantum-upgrade myaccount --hybrid
NIST-standardized
ML-DSA-65 (FIPS 204) built into the protocol — not a bolt-on. Hashing and key storage are already quantum-safe.
Hybrid mode
Require both a classical and a post-quantum signature — an attacker must break both schemes to forge.
Upgrade in place
One command rotates an account to post-quantum auth with the same address — no fund migration.
Cross-verified
Rust and TypeScript signers are proven byte-for-byte compatible against committed test vectors.
